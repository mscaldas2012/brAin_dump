The Ghost in the Machine Knows Your Schema

We keep treating LLMs like stateless functions. They're not. Understanding what an agent actually "knows" about your domain changes how you should design the boundary between code and context.

There's a design smell I keep encountering in AI-integrated systems: the team builds a beautiful, well-bounded domain model — proper aggregates, value objects, the whole DDD liturgy — and then pastes a raw JSON dump of it straight into a prompt. The implicit assumption is that the model is a blank slate that needs everything spelled out.

It isn't. And that assumption creates two opposite failure modes simultaneously.

What the model already carries

A model trained on the breadth of public software development writing has likely absorbed a reasonable prior about what an Order, a Customer, or an InvoiceLineItem probably means. Sending it a schema for these is less like teaching and more like redundant noise — noise that still consumes context window and subtly competes with your specific intent.

The interesting question isn't "what should I tell the model?" It's "what does the model already believe, and where does my domain deviate from that prior?"

Your deviations are the signal. A Shipment that represents an internal transfer rather than a customer delivery, a User that is always a legal entity rather than a person — these are the conceptual gaps worth filling. The rest is largely recoverable from context clues in your actual data.

Distribution shift is a schema problem

This reframe also clarifies what's actually happening when a prompt "breaks" after a data migration or a naming refactor. It's not a prompt bug in the traditional sense. It's a distribution shift: the model's internal representation of your domain no longer aligns with the symbols you're using. The ghost still haunts the old layout. 

-- Before refactor: model knows what "client" means here
SELECT * FROM client WHERE status = 'active'

-- After: "account" violates the model's prior
-- Same semantic, different signal strength
SELECT * FROM account WHERE status = 'active'

If you treat prompts as static configuration, this class of failure is nearly invisible until production. The fix isn't defensive prompting — it's treating your prompt layer as a domain translation boundary that needs the same change-management discipline as your API contracts.

Toward a bounded context for inference

The mental model that's been most useful for me: think of the LLM as a very knowledgeable contractor who joined mid-project. They know the industry. They've seen systems like yours. What they need from you isn't a glossary of what "order" means — they need the delta: where your system's vocabulary and semantics diverge from common practice, and why.

That constraint naturally produces tighter, more durable prompts. And it forces a useful architectural question: if you can't articulate what's unusual about your domain model, maybe the model isn't the problem.

What I'm still figuring out

The hard part is that "the model's prior" isn't inspectable in any clean way. You're reasoning about a distribution, not a fact. So the practical discipline becomes empirical: build evals, track which prompt segments actually move output quality, and treat inexplicable regressions as evidence of a schema divergence before anything else.
